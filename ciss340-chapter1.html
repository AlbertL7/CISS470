🔑 Complete Guide to Database Keys & Relationships

1. PRIMARY KEYS - The Unique Identifier

🔑

What is a Primary Key?

Definition: A primary key is a column (or combination of columns) that uniquely identifies each row in a table. No two rows can have the same primary key value, and it can NEVER be NULL.
🎯 Real-World Analogy:

Social Security Number: Just like your SSN uniquely identifies you among all US residents, a primary key uniquely identifies a row among all rows in a table.

Student ID: In a school, every student has a unique ID number. Even if two students have the same name, their ID numbers are different.

License Plate: Every car has a unique license plate number within a state - that's its primary key in the DMV database!

📋 Primary Key Rules:
Must be UNIQUE - no duplicates allowed
Cannot be NULL - must always have a value
Values should NEVER change once assigned
Every table should have one (best practice)
Only ONE primary key per table
Automatically creates an index for fast lookups

Visual Example: Primary Key in Action

employee_id PRIMARY KEY first_name last_name email hire_date
1001 John Smith john.smith@company.com 2023-01-15
1002 Sarah Johnson sarah.j@company.com 2023-02-20
1003 John Smith john.smith2@company.com 2023-03-10

↑ Notice: Two employees named "John Smith", but different employee_ids (1001 vs 1003)

💡 Why Primary Keys Matter:
  • Data Integrity: Prevents duplicate records from being inserted
  • Relationships: Other tables can reliably reference this row
  • Performance: Database creates an index automatically for fast searches
  • Updates/Deletes: Can target exactly one row without ambiguity

2. COMPOSITE PRIMARY KEYS - When One Column Isn't Enough

🔑🔑

What is a Composite Primary Key?

Definition: A composite primary key uses TWO or more columns together to create a unique identifier. The combination must be unique, even if individual values repeat.
🎯 Real-World Analogy:

Flight Identification: A flight is uniquely identified by the combination of airline + flight number + date. "AA 1234" isn't unique (flies daily), but "AA 1234 on March 15, 2024" is unique.

Apartment Number: Building 5, Apartment 2B. Neither "5" nor "2B" is unique alone, but together they identify exactly one apartment.

Class Enrollment: Student ID + Course ID together. Student 1001 can take many courses, Course MATH101 has many students, but Student 1001 in MATH101 is unique.

Visual Example: Order Line Items (Classic Composite Key Use Case)

order_id PK-1 line_number PK-2 product_id quantity price
5001 1 PROD-100 2 $29.99
5001 2 PROD-200 1 $49.99
5001 3 PROD-150 3 $19.99
5002 1 PROD-100 1 $29.99

↑ Order 5001 has 3 lines, Order 5002 has 1 line. The combination of order_id + line_number is unique.

⚠️ Important Notes:
  • BOTH columns together form ONE primary key (not two separate keys)
  • Individual columns CAN have duplicate values
  • The COMBINATION must be unique
  • BOTH columns become NOT NULL automatically

3. FOREIGN KEYS - The Relationship Builders

🔗

What is a Foreign Key?

Definition: A foreign key is a column (or columns) that creates a link between two tables. It contains values that match the primary key values in another table, establishing a parent-child relationship.
🎯 Real-World Analogy:

Emergency Contact: Your employee record has your spouse's employee ID as emergency contact. That ID is a "foreign key" pointing to their full record.

Credit Card Statement: Each transaction shows a merchant ID, which links to the merchant's full details in another system.

Bibliography Reference: A [3] in your essay points to source #3 in your bibliography - that's a foreign key relationship!

Visual Example: How Foreign Keys Connect Tables

🔍 CRITICAL: Foreign Key EXISTS IN ONLY ONE TABLE!

NO, both tables DO NOT have foreign keys! This is a common misconception.

  • Parent Table (Departments): Has ONLY its primary key, NO foreign key
  • Child Table (Employees): Has the foreign key that points to the parent
  • The Rule: In a one-to-many relationship, the foreign key is ALWAYS on the "many" side (child table)
🎯 Think of it like Phone Contacts:

Your mom doesn't need to store your phone number in her contact list to be your mom. But YOU store HER number in YOUR phone to call her. The foreign key is like storing someone else's number - only the person who needs to "call" (reference) the other table has it.

DEPARTMENTS Table (Parent - No Foreign Key Here!)
dept_id PK dept_name location
10 Sales New York
20 IT San Francisco
30 HR Chicago

↑ This table has ONLY its primary key (dept_id)

← referenced by →
EMPLOYEES Table (Child - Foreign Key is HERE!)
emp_id PK emp_name dept_id FK→DEPARTMENTS
1001 Alice Johnson 10
1002 Bob Smith 20
1003 Carol White 10
1004 David Brown 20

↑ The foreign key (dept_id) is ONLY in this table, pointing to Departments

How It Works:

  • The value "10" in Employees.dept_id matches the value "10" in Departments.dept_id
  • This creates a link: "Alice Johnson works in Sales department"
  • The foreign key ensures you can't assign an employee to department 99 (doesn't exist)
  • Multiple employees can have the same dept_id value (many employees in one department)
📋 Foreign Key Rules:
Must match an existing primary key value in the parent table (or be NULL)
Enforces "referential integrity" - can't reference non-existent data
Can have multiple foreign keys in one table
Can be NULL (unless specified as NOT NULL)
Same column can be both a primary key AND a foreign key
Updates/deletes in parent table can cascade to child tables
💡 What Foreign Keys Prevent:
  • Orphan Records: Can't have an employee in department 99 if department 99 doesn't exist
  • Broken Links: Can't delete a department that still has employees
  • Data Inconsistency: If department 10 moves from "New York" to "Boston", all employees stay connected

❌ Common Misconception: "Both Tables Have Foreign Keys"

This is WRONG! Here's why:

❌ INCORRECT Thinking:

"Departments table has a foreign key to Employees"
"Employees table has a foreign key to Departments"

This would create a circular reference and doesn't make sense!

✅ CORRECT Understanding:

Parent (Departments): Has ONLY primary key
Child (Employees): Has foreign key pointing to parent

One-way reference from child to parent!

Remember: In a relationship, only the table that needs to "look up" information in another table has the foreign key. It's like having someone's phone number - only the person who needs to make the call stores the number!

4. UNIQUE KEYS - The Alternative Identifiers

🎯

What is a Unique Key?

Definition: A unique key ensures all values in a column (or combination of columns) are different from each other. Unlike primary keys, a table can have multiple unique keys and they CAN contain NULL values (but only one NULL).
🎯 Real-World Analogy:

Email Address: In a user table, the user_id is primary key, but email must also be unique. Two users can't share an email.

Phone Number: Your phone number is unique to you, but it's not your primary identifier (that's your account number).

ISBN for Books: A book's primary key might be book_id, but ISBN is also unique and useful for lookups.

Visual Example: Multiple Unique Constraints

user_id PK username UNIQUE email UNIQUE ssn UNIQUE first_name last_name
1 johndoe john@email.com 123-45-6789 John Doe
2 janedoe jane@email.com 987-65-4321 Jane Doe
3 bobsmith bob@email.com NULL Bob Smith

↑ This table has 1 primary key and 3 unique keys. Note: SSN can be NULL for international users.

6. COMPLETE EXAMPLE - E-Commerce Database

Scenario: Online Store Database Design

Let's see how all these keys work together in a real e-commerce database:

-- Customers table with primary and unique keys
CREATE TABLE customers (
  customer_id INT PRIMARY KEY AUTO_INCREMENT,
  email VARCHAR(100) UNIQUE NOT NULL,
  phone VARCHAR(20) UNIQUE,
  first_name VARCHAR(50) NOT NULL,
  last_name VARCHAR(50) NOT NULL
);

-- Products table
CREATE TABLE products (
  product_id INT PRIMARY KEY AUTO_INCREMENT,
  sku VARCHAR(50) UNIQUE NOT NULL,
  product_name VARCHAR(200) NOT NULL,
  price DECIMAL(10,2) NOT NULL
);

-- Orders table with foreign key to customers
CREATE TABLE orders (
  order_id INT PRIMARY KEY AUTO_INCREMENT,
  customer_id INT,
  order_date DATETIME DEFAULT CURRENT_TIMESTAMP,
  total_amount DECIMAL(10,2),
  FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);

-- Order_Items with composite primary key and foreign keys
CREATE TABLE order_items (
  order_id INT,
  item_number INT,
  product_id INT,
  quantity INT NOT NULL,
  price DECIMAL(10,2) NOT NULL,
  PRIMARY KEY (order_id, item_number),
  FOREIGN KEY (order_id) REFERENCES orders(order_id),
  FOREIGN KEY (product_id) REFERENCES products(product_id)
);
🔑 Primary Keys in Action:
  • customer_id: Each customer gets unique ID (1, 2, 3...)
  • product_id: Each product gets unique ID
  • order_id: Each order gets unique ID
  • (order_id, item_number): Composite key for line items
🔗 Foreign Keys in Action:
  • orders.customer_id: Links each order to a customer
  • order_items.order_id: Links items to their order
  • order_items.product_id: Links items to products
🎯 Unique Keys in Action:
  • customers.email: No duplicate emails allowed
  • customers.phone: Each phone number unique (can be NULL)
  • products.sku: Product codes must be unique

7. EER DIAGRAMS - The Database Blueprint

📐

What is an EER Diagram?

Definition: An Enhanced Entity-Relationship (EER) diagram is a visual representation of your database structure. It shows all tables (entities), their columns (attributes), data types, keys, and most importantly, how tables relate to each other.
🎯 Real-World Analogy:

Floor Plan: Like a floor plan shows room layouts and doorways between rooms, an EER diagram shows tables and connections between them.

Family Tree: Shows relationships between family members, just like EER shows relationships between tables.

Subway Map: Shows stations (tables) and lines connecting them (relationships).

Understanding EER Symbols - The Visual Language

Table Representation:

💎 foreign_id: INT (Foreign Key)

    regular_field: VARCHAR

Key Symbols:

💎 = Foreign Key (FK)

Symbol Breakdown:

Line Endings:
  • Single Line ( | ): "One" - exactly one
  • Crow's Foot ( ⪪ ): "Many" - zero or more
  • Circle ( O ): Optional - can be zero
  • Double Line ( || ): Mandatory - must have at least one
How to Read:
  • Read from left to right AND right to left
  • "One customer can have many orders"
  • "Each order belongs to one customer"
  • The symbol closest to the table describes that table's participation

🎓 Key Pattern to Remember:

  • Parent tables (CUSTOMERS, PRODUCTS, ORDERS when parent): Have ONLY their primary keys
  • Child tables (ORDERS when child, ORDER_ITEMS): Have foreign keys pointing to parents
  • Junction tables (ORDER_ITEMS): Have multiple foreign keys to connect multiple parents
  • Direction: Foreign keys always point FROM child TO parent, never the other way
📋 How to Read an EER Diagram:
Start by identifying all primary keys (🔑) - these are the unique identifiers
Look for foreign keys (💎) - these create the relationships
Follow the relationship lines to understand how tables connect
Read relationships both ways: "One X has many Y" and "Each Y belongs to one X"
Identify junction tables (like ORDER_ITEMS) that resolve many-to-many relationships
Note data types and constraints (NN = Not Null, AI = Auto Increment)
💡 Quick Tips for Understanding EER Diagrams:

Crow's Foot = Many: Think of it like a bird's foot spreading out to grab many things

Single Line = One: A straight line represents a single connection

Foreign Keys Point Back: They're like breadcrumbs leading back to the parent table

Junction Tables: Like a sign-up sheet that connects students to classes

8. BEST PRACTICES & TIPS

✅ DO's:
Always define a primary key for every table
Use auto-increment integers for primary keys when possible
Create foreign keys to enforce relationships
Add unique constraints to natural keys (email, username, etc.)
Name keys descriptively (customer_id, not just id)
Index foreign key columns for better performance
❌ DON'Ts:
  • Don't use data that might change as primary key (like email)
  • Don't create tables without primary keys
  • Don't ignore foreign key constraints - they prevent bad data
  • Don't use spaces or special characters in key names
  • Don't make composite keys unnecessarily complex
💭 Final Memory Trick:

Primary Key = Your ID card number (one per person, never changes)

Foreign Key = Your mom's phone number in your contacts (points to her)

Unique Key = Your email address (unique but not your main ID)

Composite Key = Building + Apartment number (need both to be unique)
